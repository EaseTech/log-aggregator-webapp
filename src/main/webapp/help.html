Basic Search Criteria Section

This section lets you choose the Environment information from where you want to fetch/aggregate the logs. This is how it works :
  • A user first selects one of the available environments.
  • He then selects the available Applications
  • Next he selects one or more components.
    NOTE: A user can select more than one components, in which case the Logging Framework will combine the results into one.
    NOTE: For any action that needs to be performed the user has to provide values for the above criteria.
  • Finally, a user can then decide to generate one of the 3 available reports :
    1. Unique Exception Names report : which fetches the Unique Names of the Exceptions that occurred in the selected components.
    2. Exception and Count report : which fetches the Unique Names of the Exceptions that occurred in the selected components and how many times the exception occurred.
    3. Truncated Exception report : which fetches the Truncated Exceptions that occurred in the selected components

Extended Search Criteria Section

This section allows a user to enter any arbitrary expression in the input box and search for it in the log files of the selected components. A user can provide either a constant expression like a Request Id (VDX-AHX-WEF) or a regular expression. In case the user wants to search based on a regular expression, he should tick the is regular expression checkbox.

Fetch Exception Trace

Fetch Exception Trace is a quick mechanism to fetch the exception stack trace given an exception name. The exception is searched for in the log files of all the selected components and then shown to the end user. A user has the following extra options to refine the search for an exception trace:
  1. Start Date : A user can specify the start date/time for searching the exception. It defaults to todays date. The accepted formats are
    1. yyyy-mm-dd
    2. yyyy-mm-dd HH
    3. yyyy-mm-dd HH:mm
    4. yyyy-mm-dd HH:mm:ss
    5. yyyy-mm-dd HH:mm:ss.SSS
  2. End Date : A user can specify the end date/time for stopping the search for the exception. It defaults to now. The accepted formats are same as above.
  3. Lines Before Exception : How many lines should be fetched from the logs BEFORE the exception was logged. Defaults to ZERO.
  4. Lines After Exception : How many lines should be fetched from the logs AFTER the exception was logged. Defaults to 100.
